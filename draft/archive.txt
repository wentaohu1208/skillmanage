================================================================================
                          Archive Area 详解
================================================================================

定位

  Active:    正在用的skill，参与检索
  Archive:   曾经有用但现在不活跃的skill，不参与正常检索，可被召回
  Forgotten: 彻底不用了，只留名字做去重

  Archive只存放曾经在Active中验证过有价值的skill。
  不是任何skill都能进Archive，必须是"曾经的好skill"。

  唯一入口: Active降级（importance低于θ_archive连续T1轮）
  唯一出口: 召回回Active 或 降级到Forgotten


================================================================================
一、数据结构
================================================================================

  @dataclass
  class ArchivedSkill:
      skill_summary: str                # 压缩后的skill描述
      original_skill_id: str            # 原始skill id
      original_skill_full: Skill        # 原始完整版本的备份（用于召回恢复）
      archived_at: int                  # 归档时的round
      last_used_at: int                 # 归档前最后一次使用的round
      recall_count: int                 # 被召回并成功的次数
      inactive_rounds: int              # 归档后连续未被召回的轮数


================================================================================
二、进入Archive
================================================================================

  Active中Skill S的importance < θ_archive 连续T1轮
      │
      ▼
  从Active中移除S
      │
      ▼
  保存S的完整版本（original_skill_full）
      │
      ▼
  压缩S的描述（用LLM精炼）
      │
      ▼
  存入Archive

--------------------------------------------------------------------------------
压缩示例
--------------------------------------------------------------------------------

  原始 (150 tokens):
    name: "temperature_change_object"
    steps: [
      "1. 判断需要加热还是冷却",
      "2. 如果加热，go to microwave，执行heat命令",
      "3. 如果冷却，go to fridge，执行cool命令",
      "4. 等待操作完成",
      "5. 确认物品状态已改变"
    ]
    warnings: ["先确认microwave/fridge是否为空"]

  压缩后 (40 tokens):
    summary: "加热用microwave/heat，冷却用fridge/cool，先确认设备为空"

  完整版本保留在original_skill_full中，召回时恢复。


================================================================================
三、Archive的检索触发
================================================================================

  Archive不参与agent的正常检索流程。

  正常流程：
    Agent接到task → 从Active中检索top-K skill → 执行任务
    Archive不在这个流程里。

  Archive被触发的唯一情况：Active中检索不到合适的skill。

--------------------------------------------------------------------------------
触发流程
--------------------------------------------------------------------------------

  Agent接到task
      │
      ▼
  从Active中检索 → top-K skill的最高相似度
      │
      ├─ 最高sim > threshold → 正常用Active的skill → Archive不参与
      │
      └─ 最高sim < threshold → Active中没有好的匹配
          │
          ▼
      去Archive中搜索
          │
          ▼
      用task embedding和Archive中每个skill_summary的embedding匹配
          │
          ├─ 找到匹配（sim > threshold）→ 触发召回
          │
          └─ 没找到 → agent裸跑

  这样Archive只在"Active帮不了agent"的时候才被翻出来。


================================================================================
四、召回流程
================================================================================

  Archive中Skill S被检索命中
      │
      ▼
  恢复S的完整版本（original_skill_full）
      │
      ▼
  用完整版本拼入prompt，agent执行任务
      │
      ▼
  记录结果
      │
      ├─ 成功 → recall_count += 1, inactive_rounds = 0
      │         │
      │         ▼
      │     recall_count ≥ R ?
      │         │
      │     ┌───┴───┐
      │    Yes      No
      │     │       │
      │     ▼       ▼
      │  Promote   留在Archive
      │  回Active   等下次召回
      │  (重置meta)
      │
      └─ 失败 → 留在Archive, inactive_rounds = 0
                （虽然失败了，但有人需要这个skill，重置inactive计数）

--------------------------------------------------------------------------------
为什么失败也重置inactive_rounds
--------------------------------------------------------------------------------

  被召回说明有相关任务出现了，只是这个skill没帮上忙。
  任务分布确实shift回来了，只是skill效果不好。
  不应该因为一次失败就加速遗忘。

--------------------------------------------------------------------------------
为什么需要成功R次才回Active
--------------------------------------------------------------------------------

  Skill在Archive里待了很久，环境可能变了。
  一次成功可能是偶然的。
  多次成功才能确认它确实还有用。

  R=2就够了：
    第1次成功 → 可能是偶然
    第2次成功 → 大概率确实有用 → 回Active

--------------------------------------------------------------------------------
Promote回Active时的处理
--------------------------------------------------------------------------------

  recall_count ≥ R
      │
      ▼
  恢复完整版本 → 放入Active
  重置meta：
    call_count = 0
    success_count = 0
    success_rate = 0
    last_used_at = current_round
    avg_reward = 0

  从Archive中移除该skill

  注意：meta完全重置，因为旧的meta已经过时了。
  回到Active后重新积累统计数据。


================================================================================
五、通向Forgotten
================================================================================

  Archive中Skill S长期无人触发召回

  每轮: inactive_rounds += 1
      │
      ▼
  inactive_rounds ≥ max_archive_inactive (如50轮)?
      │
      ├─ No → 继续留在Archive
      │
      └─ Yes → 降级到Forgotten
                │
                ▼
            只保留:
              skill_id
              name
              一句话summary（来自Archive的压缩版）
            丢弃:
              完整版本（original_skill_full）
              所有meta


================================================================================
六、完整例子
================================================================================

  Round 1-100:
    Skill "heat_object" 在Active中，经常被pick_heat任务使用
    meta: call_count=25, success_rate=0.88

  Round 100-142:
    任务分布变了，全是pick_clean任务
    heat_object不再被使用

    Round 140: importance=0.28 < θ_archive(0.3) → 第1轮低于阈值
    Round 141: importance=0.25 → 第2轮
    Round 142: importance=0.22 → 第3轮 = T1
    → 降级到Archive
    → 压缩: 150 tokens → 40 tokens summary
    → 保留完整版本备份

  Round 142-179:
    在Archive中，没有heat相关任务出现
    inactive_rounds: 1, 2, 3, ..., 37

  Round 180:
    突然来了一个pick_heat任务
    Active中没有heat相关skill → 最高sim < threshold
    → 搜索Archive → 命中heat_object
    → 恢复完整版 → 拼入prompt → agent执行 → 成功 ✓
    → recall_count=1, inactive_rounds=0（重置）

  Round 190:
    又一个pick_heat任务
    Active中仍没有heat skill → Archive命中 → 召回 → 成功 ✓
    → recall_count=2 ≥ R(2)
    → Promote回Active → 重置meta → 从Archive移除

    Active中重新有了heat_object

  ─────────────────────────────────────────────────────
  另一种结局：如果Round 180之后再也没有heat任务

  Round 180: 召回成功一次，recall_count=1, inactive_rounds=0
  Round 181-230: 再也没被触发
    inactive_rounds: 1, 2, ..., 50
  Round 230: inactive_rounds=50 ≥ max_archive_inactive(50)
    → 降级到Forgotten
    → 只保留 {id, name: "heat_object", summary: "用microwave加热物品"}
    → 完整版本被丢弃


================================================================================
七、Archive的价值
================================================================================

  没有Archive的情况：
    Active降级 → 直接Forgotten → 永久丢失
    任务分布shift回来时 → 需要重新从trajectory学习
    → 浪费：同样的skill要重新走Acquisition全流程
      （积累M条记录 → Alignment → Formalization → 进Active）

  有Archive的情况：
    Active降级 → Archive保留完整版本 → 任务分布shift回来 → 快速召回
    → 省了重新学习的时间和API开销

  Archive本质上是对non-stationary task distribution的保险机制。


================================================================================
八、和Active的联动
================================================================================

  ┌──────────────────────────────────────────────────────┐
  │                     ACTIVE                            │
  │                                                        │
  │  检索：每个task从Active中取top-K skill                  │
  │  Active中sim < threshold →                            │
  │      │                                                 │
  │      └→ 搜索Archive                                   │
  │                                                        │
  │  降级：importance低 连续T1轮 → 送到Archive              │
  │  接收：Archive召回成功R次 → 重新进入Active              │
  └────────────────────┬───────────────┬──────────────────┘
                       │               ↑
                importance低       召回成功R次
                       │               │
                       ▼               │
                ┌──────────────────────┘
                │     ARCHIVE
                │
                │  存储：压缩版summary + 完整版备份
                │  触发：Active检索不到时才搜索Archive
                │  召回：恢复完整版 → 执行 → 成功R次 → 回Active
                │  清理：inactive ≥ max轮 → Forgotten
                └──────────────┬──────────────────────────┘
                               │
                          长期无人召回
                               │
                               ▼
                        ┌──────────────┐
                        │   Forgotten  │
                        └──────────────┘


================================================================================
九、调API的地方
================================================================================

  ┌────────────────────────────┬──────────┬──────────────────────────┐
  │ 操作                        │ 调API？  │ 频率                      │
  ├────────────────────────────┼──────────┼──────────────────────────┤
  │ 进入Archive时压缩skill描述  │ ✅       │ 极少，skill降级时          │
  │ Archive检索（embedding匹配）│ ❌       │ -（本地cosine similarity） │
  │ 召回时恢复完整版本          │ ❌       │ -（从备份中读取）          │
  │ inactive_rounds计数        │ ❌       │ -（计数器）                │
  └────────────────────────────┴──────────┴──────────────────────────┘

  Archive的API开销几乎为零。唯一调API的是降级时的LLM压缩，但这极少发生。


================================================================================
十、超参数
================================================================================

  参数                     含义                          建议范围
  ────────────────────────────────────────────────────────────────
  R                        召回回Active需要的成功次数      2~3
  max_archive_inactive     降级到Forgotten的不活跃轮数     40~60
