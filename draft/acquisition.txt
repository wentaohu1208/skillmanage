================================================================================
                    Acquisition 详解（增量式设计）
================================================================================

核心问题

  两类数据集，skill的"原材料"不同：

  ALFWorld/WebShop:  原材料是action trajectory（操作序列）
  MATH/BBH:          原材料是reasoning chain（推理链）

  但两者都要走同一套Acquisition流程 → 需要统一框架


统一视角

  不管是trajectory还是CoT，本质都是一个成功的step序列：

  ALFWorld trajectory:
    step1: go to countertop 1
    step2: take apple 1 from countertop 1
    step3: go to microwave 1
    step4: heat apple 1 with microwave 1
    step5: go to shelf 1
    step6: put apple 1 in/on shelf 1

  MATH CoT:
    step1: 这是一个二次方程 x²+5x+6=0
    step2: 尝试因式分解，找两个数乘积为6，和为5
    step3: 2×3=6, 2+3=5，符合
    step4: 因式分解为 (x+2)(x+3)=0
    step5: x=-2 或 x=-3

  统一抽象：都是 [step1, step2, ..., stepN] → 成功


================================================================================
Step 1: 收集判断 — 什么时候收集、收集什么
================================================================================

  Agent完成一个task
      │
      ├─ 成功 → 成功路径（提取skill的steps）
      │    │
      │    ├─ 没用任何skill（裸跑成功）→ 一定收集整条trajectory ★
      │    │
      │    └─ 用了skill成功 → 检查有没有新pattern
      │         │
      │         ├─ trajectory中有skill未覆盖的新片段 → 只收集新片段
      │         │
      │         └─ trajectory完全被已有skill覆盖 → 不收集，只更新meta
      │
      └─ 失败 → 失败路径（提取warnings）
           │
           ├─ 进度太低（agent执行步数 < 2步）→ 跳过
           │
           └─ 进度足够（≥ 2步）→ LLM分析失败原因 → 提取warning
                │
                ├─ 匹配到已有Active skill → 附加到该skill的warnings
                │
                └─ 没匹配到 → 有行动指导？
                     │
                     ├─ 有 → 转化为独立skill（confidence低）进Active
                     │
                     └─ 没有 → 丢弃


--------------------------------------------------------------------------------
怎么判断"trajectory是否被已有skill覆盖"
--------------------------------------------------------------------------------

  系统天然知道agent用了哪些skill（因为检索是系统做的）。

  记录：
    used_skills = [Skill_S7]            ← 系统在检索阶段记录
    trajectory = [step1, ..., step6]    ← 系统在执行阶段记录

  用LLM判断覆盖情况：

  Prompt:
    Agent使用了以下skill完成任务：
    Skill: "{skill_description}"
    步骤: {skill_steps}

    Agent实际执行的trajectory：
    {trajectory}

    请判断：
    1. trajectory中哪些步骤是在执行这个skill？
    2. 哪些步骤是skill未覆盖的、agent自己额外做的？

  LLM输出:
    skill覆盖的步骤: step3, step4, step5, step6
    未覆盖的步骤: step1(导航到物品位置), step2(拿取物品)

  判断逻辑：

  覆盖率 = 被skill覆盖的步骤数 / 总步骤数

  if 覆盖率 == 100%:
      → 完全覆盖，不收集，只更新meta

  elif 覆盖率 > 50%:
      → 部分覆盖，只收集未覆盖的步骤

  elif 覆盖率 ≤ 50%:
      → 大部分是新的，收集整条trajectory


  交互式任务例子（ALFWorld）：

    Agent用了skill "找物品并加热"（覆盖step3-6）
    完整trajectory: [step1, step2, step3, step4, step5, step6]
    skill覆盖:                    [step3, step4, step5, step6]
    未覆盖: [step1, step2]

    覆盖率 = 4/6 = 67% > 50% → 只收集未覆盖的 [step1, step2]


  推理任务例子（MATH）：

    Agent用了skill "因式分解法"（覆盖step2-4）
    完整CoT: [step1, step2, step3, step4, step5]
    skill覆盖:       [step2, step3, step4]
    未覆盖: [step1] 和 [step5]

    覆盖率 = 3/5 = 60% > 50% → 只收集 [step1, step5]


================================================================================
Step 2: Segmentation（切分）
================================================================================

  把收集到的执行记录切分为有意义的片段。
  你的意思是Segmentation（切分）只在没覆盖的区域进行?                                                                                           

⏺ 不是。Segmentation对整条收集到的内容都做切分。
                                                                                                                                  
  但"收集到的内容"因覆盖率不同而不同：                                                                                                          
                                                                                                                                                
  覆盖率100% → 不收集 → 不做Segmentation                                                                                                        
  覆盖率>50%  → 只收集未覆盖的片段 → 只对未覆盖片段做Segmentation                                                                               
  覆盖率≤50%  → 收集整条trajectory → 对整条做Segmentation                                                                                       
  裸跑成功    → 收集整条trajectory → 对整条做Segmentation   

--------------------------------------------------------------------------------
交互式任务（ALFWorld/WebShop）— 按子目标切分
--------------------------------------------------------------------------------

  ALFWorld trajectory:
    [go to countertop 1, take apple 1, go to microwave 1, heat apple 1,
     go to shelf 1, put apple 1]

  切分依据：环境状态的显著变化
    状态变化1: 手中从空 → 持有apple      → 边界
    状态变化2: apple从常温 → 被加热       → 边界
    状态变化3: apple从手中 → 放到shelf    → 边界

  切分结果：
    片段A: [go to countertop, take apple]           → "获取目标物品"
    片段B: [go to microwave, heat apple]             → "加热物品"
    片段C: [go to shelf, put apple]                  → "放置物品"


  WebShop trajectory:
    [search "red pillow", click "Sort by price", click "Item 3",
     click "Size: Large", click "Buy"]

  切分依据：页面类型变化
    搜索页 → 结果列表页 → 商品详情页 → 购买完成页

  切分结果：
    片段A: [search "red pillow"]                     → "搜索商品"
    片段B: [click "Sort by price", click "Item 3"]   → "排序并选择"
    片段C: [click "Size: Large", click "Buy"]        → "选属性并购买"

--------------------------------------------------------------------------------
推理任务（MATH/BBH）— 按推理阶段切分
--------------------------------------------------------------------------------

  MATH CoT:
    [识别题型, 设未知数, 列方程, 因式分解, 求解, 验证]

  切分依据：推理目的的变化

  切分结果：
    片段A: [识别题型]                    → "问题分类"
    片段B: [设未知数, 列方程]            → "建模"
    片段C: [因式分解, 求解]             → "求解技巧"
    片段D: [验证]                       → "验证答案"

--------------------------------------------------------------------------------
实现：统一用LLM做切分
--------------------------------------------------------------------------------

  Prompt:
    以下是一个成功的{任务类型}执行记录：
    {trajectory}

    请将其切分为若干有意义的片段，每个片段完成一个独立的子目标。
    对每个片段，给出：
    1. 包含的步骤
    2. 这个片段完成的子目标（一句话描述）


================================================================================
Step 3: Incremental Alignment（增量式跨记录对齐）
================================================================================

  目的：从多条同类记录中找共同pattern，过滤偶然成功。

--------------------------------------------------------------------------------
前提：知道哪些task属于"同一类"
--------------------------------------------------------------------------------

  ALFWorld:  自带标签 → pick_heat_then_place都是同一类
  MATH:      自带标签 → Algebra Level 3都是同一类
  BBH:       自带标签 → boolean_expressions都是同一类
  WebShop:   无标签 → 需要自己分类

  WebShop的分类方法：

  方法1: LLM打标签
    输入: "I need a red throw pillow under $30"
    LLM输出: "家居/装饰"

  方法2: embedding聚类
    所有指令 → embedding向量 → K-means聚类 → 得到类别

--------------------------------------------------------------------------------
为什么用增量式而非批量式
--------------------------------------------------------------------------------

  批量式的问题：

    等攒够N条同类记录 → 一次性做Alignment → 提取skill

    问题1: 前N-1条记录期间完全没有skill产出，agent只能裸跑
    问题2: N的选择直接影响confidence质量
           N太小 → confidence不可靠
           N太大 → 等太久

  增量式的优势：

    每来一条新记录 → 和已有记录对比 → 动态更新pattern和confidence
    达到最低门槛M条就可以开始提取
    confidence随记录增加持续变准

  对比：
  ┌──────────┬────────────────────────┬──────────────────────────┐
  │          │ 批量式                  │ 增量式                    │
  ├──────────┼────────────────────────┼──────────────────────────┤
  │ 触发时机  │ 攒够N条后一次性         │ 每来一条新记录就更新       │
  │ 等待时间  │ 长（前N-1条空转）       │ 短（达到M条就开始提取）    │
  │ confidence│ 固定（基于N条一次计算）  │ 动态（随记录增加持续更新） │
  │ skill质量 │ 取决于N的选择           │ 早期质量一般，后期越来越准  │
  └──────────┴────────────────────────┴──────────────────────────┘

--------------------------------------------------------------------------------
核心数据结构：PatternBuffer
--------------------------------------------------------------------------------

  每类任务维护一个PatternBuffer：

  @dataclass
  class PatternBuffer:
      task_type: str                          # 任务类别
      records: List[SegmentedTrajectory]      # 已收集的切分记录
      pattern_counts: Dict[str, int]          # 每个pattern出现次数
      total_records: int                      # 总记录数
      extracted_patterns: Set[str]            # 已提取为skill的pattern

--------------------------------------------------------------------------------
增量式Alignment流程
--------------------------------------------------------------------------------

  每来一条新的成功记录：

  1. Segmentation切分为片段

  2. 放入对应task_type的PatternBuffer

  3. 和已有patterns做语义匹配
     - 匹配到已有pattern → count += 1
     - 没匹配到 → 加入候选，count = 1

  4. 更新每个pattern的confidence = count / total_records

  5. 检查每个pattern是否满足提取条件


  提取条件（两个都要满足）：

    total_records >= M    最少记录数（如M=3），防止样本太少
    confidence >= r       最低复现率（如r=0.5），过滤偶然pattern
    pattern未被提取过      防止重复提取

--------------------------------------------------------------------------------
增量式Alignment具体例子
--------------------------------------------------------------------------------

  MATH Algebra Level 3:

  Round 5: 第1条成功CoT
    切分: [识别二次方程] [因式分解] [求解]
    PatternBuffer更新:
      "识别二次方程" count=1  confidence=1/1=1.0
      "因式分解"     count=1  confidence=1/1=1.0
      "求解"         count=1  confidence=1/1=1.0
    total_records=1 < M(3) → 不提取任何skill

  Round 18: 第2条成功CoT
    切分: [识别二次方程] [配方法] [求解]
    PatternBuffer更新:
      "识别二次方程" count=2  confidence=2/2=1.0
      "因式分解"     count=1  confidence=1/2=0.5
      "配方法"       count=1  confidence=1/2=0.5  (新pattern)
      "求解"         count=2  confidence=2/2=1.0
    total_records=2 < M(3) → 仍不提取

  Round 33: 第3条成功CoT
    切分: [识别二次方程] [因式分解] [求解] [验证]
    PatternBuffer更新:
      "识别二次方程" count=3  confidence=3/3=1.0  ★
      "因式分解"     count=2  confidence=2/3=0.67 ★
      "配方法"       count=1  confidence=1/3=0.33
      "求解"         count=3  confidence=3/3=1.0  (太通用，后面会过滤)
      "验证"         count=1  confidence=1/3=0.33
    total_records=3 >= M(3) → 开始检查提取条件

    "识别二次方程" confidence=1.0 >= r(0.5) → 提取！进入Formalization
    "因式分解"     confidence=0.67 >= r(0.5) → 提取！
      注意：有变体（因式分解 vs 配方法）→ 参数化为"解法={因式分解|配方法}"
    "配方法"       confidence=0.33 < r(0.5) → 不提取（但已被因式分解的变体覆盖）
    "求解"         confidence=1.0 >= r(0.5) → 太通用，过滤
    "验证"         confidence=0.33 < r(0.5) → 不提取

  Round 60: 第4条成功CoT
    切分: [识别二次方程] [求根公式] [求解]
    PatternBuffer更新:
      "识别二次方程" count=4  confidence=4/4=1.0
      "因式分解"     count=2  confidence=2/4=0.5  ← 下降了
      "配方法"       count=1  confidence=1/4=0.25
      "求根公式"     count=1  confidence=1/4=0.25 (新变体)
      ...

    已提取的"因式分解"skill的confidence从0.67更新为0.5
    → 该skill在Active中 → 更新其meta中的confidence

    "求根公式"是新变体 → 更新已提取skill的参数：
      解法={因式分解|配方法|求根公式}

--------------------------------------------------------------------------------
"太通用"pattern的过滤
--------------------------------------------------------------------------------

  有些pattern复现率很高但没有信息量，例如"求解"在所有数学题中都出现。

  过滤方法：跨类别通用性检查

    如果一个pattern在多个不同类别中都高频出现
    → 说明它不是某类任务特有的skill，而是通用操作
    → 不提取

    例：
      "求解" 在Algebra中出现100%，在Geometry中也100%，在Number Theory中也100%
      → 跨类别通用 → 不提取

      "因式分解" 在Algebra中出现67%，在Geometry中出现5%，在Number Theory中出现10%
      → 主要属于Algebra → 提取为Algebra的skill

  判断标准：
    如果pattern在目标类别中的confidence显著高于其他类别 → 提取
    如果pattern在所有类别中都差不多 → 不提取（太通用）


================================================================================
Step 4: Formalization（形式化）
================================================================================

  把满足提取条件的pattern转化为标准skill格式。

  Alignment输出:
    pattern: "识别问题类型并选择对应解法"
    复现率: 3/3 = 1.0
    变体: 解法={因式分解, 配方法}
    来源: algebra_level3

          ↓ LLM泛化

  标准skill格式:
    {
      "name": "quadratic_equation_solver",
      "description": "识别二次方程并选择合适的解法",
      "precondition": "表达式包含x²项",
      "parameters": ["method: factoring | completing_square | quadratic_formula"],
      "steps": [
        "1. 确认方程为ax²+bx+c=0的形式",
        "2. 优先尝试因式分解（简单情况）",
        "3. 因式分解不行则用配方法或求根公式",
        "4. 代回原方程验证"
      ],
      "source": "experience_distillation",
      "task_type": "algebra_level3",
      "confidence": 1.0,
      "token_cost": 95,
      "warnings": []
    }

  Formalization的Prompt：

  Prompt:
    以下是从多条成功记录中提取的共同pattern：

    pattern: {pattern_description}
    复现率: {rate}
    变体: {variants}
    来源任务类型: {task_type}

    请将其转化为一个通用skill，要求：
    1. name: 简短英文命名
    2. description: 一句话描述，说明什么情况下使用
    3. precondition: 使用这个skill的前提条件
    4. steps: 具体步骤（泛化表述，不含具体数值）
    5. 如果有变体，用参数化表示

    要求描述尽量精炼，控制在100 tokens以内。

  Confidence传递：
    Formalization后的skill带着alignment阶段的confidence直接进入Active。
    后续新记录到来时，confidence会持续更新。


================================================================================
从失败轨迹中提取Warnings
================================================================================

  成功轨迹告诉agent：什么该做（steps）
  失败轨迹告诉agent：什么不该做（warnings）

  两者互补，一个skill同时包含正面指导和负面警告。

--------------------------------------------------------------------------------
什么样的失败值得分析
--------------------------------------------------------------------------------

  不是所有失败都有价值：

  ❌ 不分析：agent完全没思路（第1步就卡住）→ 没有有用信息
  ❌ 不分析：超时/环境错误导致的失败 → 不是策略问题
  ✅ 分析：agent有一定进展但最终失败（执行步数≥2步）
           → "差一点成功"的失败最有价值

--------------------------------------------------------------------------------
失败分析Prompt
--------------------------------------------------------------------------------

  Prompt:
    Agent尝试完成以下任务但失败了：
    任务: {task}
    执行记录: {trajectory}
    失败点: {where_it_failed}

    请分析：
    1. 失败的根本原因是什么？
    2. 下次遇到类似任务应该避免什么？（一句话，具体可操作）
    3. 这个教训适用于哪类任务？

--------------------------------------------------------------------------------
Warning处理（简化版，不用WarningBuffer）
--------------------------------------------------------------------------------

  失败（执行步数≥2步）→ LLM分析 → 提取warning
                          │
                          ▼
                  匹配已有Active skill？
                          │
                  ┌───────┴───────┐
                 Yes              No
                  │                │
                  ▼                ▼
            附加到该skill     有行动指导？
            的warnings中          │
                            ┌─────┴─────┐
                           Yes          No
                            │            │
                            ▼            ▼
                       转化为skill      丢弃
                       (confidence低)
                       进Active

  "有行动指导"的判断标准：
    ✅ "找食物时优先搜索countertop和fridge，不要搜索shelf" → 有具体动作建议
    ✅ "系数简单时优先因式分解，不要直接用求根公式" → 有方法选择建议
    ❌ "agent在第5步卡住了" → 只描述现象，没说怎么办
    ❌ "这道题很难" → 没有任何有用信息

--------------------------------------------------------------------------------
Warning附加到已有skill的例子
--------------------------------------------------------------------------------

  已有Active skill:
    {
      "name": "find_object",
      "steps": ["推断物品可能的位置", "依次搜索"],
      "warnings": []
    }

  失败轨迹分析得到：
    "食物类物品通常在countertop或fridge，不在shelf/drawer/cabinet"

  匹配到"find_object" → 附加：
    {
      "name": "find_object",
      "steps": ["推断物品可能的位置", "依次搜索"],
      "warnings": [
        "食物类物品优先搜索countertop和fridge，避免shelf/drawer/cabinet"
      ]
    }

--------------------------------------------------------------------------------
Warning转化为独立skill的例子
--------------------------------------------------------------------------------

  失败轨迹分析得到：
    "解二次方程时，系数简单优先因式分解，不要直接用求根公式"

  Active中没有匹配的skill → 检查有无行动指导 → 有

  转化为skill:
    {
      "name": "quadratic_method_selection",
      "description": "选择二次方程的解法策略",
      "steps": [
        "1. 检查系数是否简单（小整数）",
        "2. 简单系数 → 优先因式分解",
        "3. 复杂系数 → 用求根公式"
      ],
      "warnings": [],
      "source": "failure_analysis",
      "confidence": 0.5
    }

  注意：confidence设为较低值（0.5），因为来自失败推断，未经成功验证。

--------------------------------------------------------------------------------
Agent使用skill时看到的完整prompt
--------------------------------------------------------------------------------

  Skill: 解二次方程
  步骤：
    1. 确认方程为ax²+bx+c=0的形式
    2. 优先尝试因式分解（简单情况）
    3. 因式分解不行则用配方法或求根公式
    4. 代回原方程验证
  注意：
    - 系数简单时优先因式分解，不要直接用求根公式（容易算错判别式）
    - 求解后务必代回原方程验证


================================================================================
完整的增量式Acquisition Pipeline（含失败学习）
================================================================================

  Agent完成task
      │
      ├─ 成功 → 成功路径
      │         │
      │         ▼
      │     用了skill吗？
      │         │
      │         ├─ 没用（裸跑成功）→ 收集整条trajectory
      │         │
      │         └─ 用了skill → LLM判断覆盖率
      │              │
      │              ├─ 覆盖率100% → 不收集，只更新已用skill的meta
      │              ├─ 覆盖率>50%  → 只收集未覆盖的片段
      │              └─ 覆盖率≤50%  → 收集整条trajectory
      │         │
      │         ▼
      │     Segmentation → PatternBuffer → 检查提取条件
      │         │
      │         ▼ (满足条件时)
      │     Formalization → 查Forgotten去重 → 进入Active
      │
      │
      └─ 失败 → 失败路径
                │
                ▼
            执行步数 ≥ 2步？
                │
            ┌───┴───┐
           No      Yes
            │       │
            ▼       ▼

================================================================================
Confidence的持续更新机制
================================================================================

  Confidence不是一次性的，而是随新记录不断更新。

  Round 33: skill "因式分解" 被提取，confidence=2/3=0.67
  Round 60: 第4条记录，该pattern没出现 → confidence=2/4=0.50
  Round 85: 第5条记录，该pattern出现了 → confidence=3/5=0.60

  更新传递：
    skill在Active中 → 更新其meta中的confidence
    如果confidence跌破r → 标记为"质量存疑"（但不自动移除，由importance score决定去留）

  变体发现也是增量的：
    Round 33: 解法={因式分解, 配方法}
    Round 60: 发现新变体"求根公式" → 解法={因式分解, 配方法, 求根公式}
    → 更新Active中对应skill的参数列表


================================================================================
关键超参数
================================================================================

  M     最少记录数          建议值: 3       太小不可靠，太大等太久
  r     最低复现率阈值      建议值: 0.5     过滤偶然pattern
  D     Forgotten去重衰减   建议值: 100轮   超过D轮允许重新学习
  P     失败分析步数阈值    建议值: 2       执行步数≥2步的失败才分析

  这四个超参数可以在实验中做sensitivity analysis。

### 一步一步走一遍

假设你在跑MATH的Algebra Level 3的题目。PatternBuffer初始为空。

---

**Round 5：第1道Algebra Level 3题成功了**

题目："解方程 x²+5x+6=0"

Agent的CoT：
```
识别这是二次方程 → 因式分解 → 得到(x+2)(x+3)=0 → 求解x=-2或x=-3
```

Segmentation切分为：
```
片段1: "识别二次方程"
片段2: "因式分解"
片段3: "求解"
```

放入PatternBuffer：
```
PatternBuffer (algebra_level3):
  total_records = 1

  patterns:
    "识别二次方程"  出现1次  confidence = 1/1 = 1.0
    "因式分解"      出现1次  confidence = 1/1 = 1.0
    "求解"          出现1次  confidence = 1/1 = 1.0
```

**检查提取条件：total_records=1 < M(3) → 不提取。才1条记录，说明不了什么。**

---

**Round 18：第2道Algebra Level 3题成功了**

题目："解方程 x²-4x+4=0"

Agent的CoT：
```
识别这是二次方程 → 配方法 → 得到(x-2)²=0 → 求解x=2
```

Segmentation切分为：
```
片段1: "识别二次方程"
片段2: "配方法"
片段3: "求解"
```

放入PatternBuffer，和已有patterns做匹配：
```
"识别二次方程" → 匹配到已有pattern → count从1变成2
"配方法"       → 没匹配到（和"因式分解"不同）→ 新增
"求解"         → 匹配到已有pattern → count从1变成2
```

PatternBuffer更新为：
```
PatternBuffer (algebra_level3):
  total_records = 2

  patterns:
    "识别二次方程"  出现2次  confidence = 2/2 = 1.0
    "因式分解"      出现1次  confidence = 1/2 = 0.5   ← 下降了！
    "配方法"        出现1次  confidence = 1/2 = 0.5   ← 新的
    "求解"          出现2次  confidence = 2/2 = 1.0
```

注意"因式分解"的confidence从1.0变成0.5。**因为第2条记录没用因式分解，用了配方法。** 所以"因式分解"不是每次都出现的。

**检查提取条件：total_records=2 < M(3) → 还是不提取。**

---

**Round 33：第3道Algebra Level 3题成功了**

题目："解方程 2x²+7x+3=0"

Agent的CoT：
```
识别这是二次方程 → 因式分解 → 得到(2x+1)(x+3)=0 → 求解x=-0.5或x=-3 → 验证
```

Segmentation切分为：
```
片段1: "识别二次方程"
片段2: "因式分解"
片段3: "求解"
片段4: "验证"
```

PatternBuffer更新：
```
PatternBuffer (algebra_level3):
  total_records = 3

  patterns:
    "识别二次方程"  出现3次  confidence = 3/3 = 1.0
    "因式分解"      出现2次  confidence = 2/3 = 0.67
    "配方法"        出现1次  confidence = 1/3 = 0.33
    "求解"          出现3次  confidence = 3/3 = 1.0
    "验证"          出现1次  confidence = 1/3 = 0.33   ← 新的
```

**现在total_records=3 ≥ M(3)，开始逐个检查！**

```
"识别二次方程"  confidence=1.0  ≥ r(0.5)  → ✅ 满足
"因式分解"      confidence=0.67 ≥ r(0.5)  → ✅ 满足
"配方法"        confidence=0.33 < r(0.5)  → ❌ 不够
"求解"          confidence=1.0  ≥ r(0.5)  → ✅ 但太通用（每类题都有"求解"）→ 过滤
"验证"          confidence=0.33 < r(0.5)  → ❌ 不够
```

**提取2个skill：**
- "识别二次方程" → Formalization → 进Active
- "因式分解" → 但发现它和"配方法"是同一步骤的不同变体 → 参数化为"解二次方程(方法={因式分解|配方法})" → Formalization → 进Active

---

**Round 60：第4道Algebra Level 3题成功了**

题目："解方程 x²+2x+5=0"

Agent的CoT：
```
识别这是二次方程 → 判别式<0用求根公式 → 得到复数解 → 求解
```

Segmentation切分为：
```
片段1: "识别二次方程"
片段2: "求根公式"       ← 新变体！
片段3: "求解"
```

PatternBuffer更新：
```
PatternBuffer (algebra_level3):
  total_records = 4

  patterns:
    "识别二次方程"  出现4次  confidence = 4/4 = 1.0
    "因式分解"      出现2次  confidence = 2/4 = 0.5   ← 又下降了
    "配方法"        出现1次  confidence = 1/4 = 0.25
    "求根公式"      出现1次  confidence = 1/4 = 0.25   ← 新变体
    "求解"          出现4次  confidence = 4/4 = 1.0
    "验证"          出现1次  confidence = 1/4 = 0.25
```

**两件事发生：**

1. 之前提取的"因式分解"skill，confidence从0.67降到了0.5
   - 该skill在Active中 → 更新其meta中的confidence
   
2. 发现"求根公式"是新变体 → 更新已提取skill的参数：
   - 方法={因式分解|配方法|求根公式}

---

### 整个过程的关键点

```
Round 5:  1条记录 → 看不出什么 → 等
Round 18: 2条记录 → 开始看出一些趋势 → 但还不够 → 等
Round 33: 3条记录 → 够了，pattern清晰了 → 提取
Round 60: 4条记录 → 更新confidence，发现新变体

不是等到某个时刻一次性分析
而是每来一条记录就更新一次，随时可能触发提取
```


================================================================================
第一阶段（Task Stream）中调API的地方
================================================================================

  一个task的完整流程，标注每一步是否调API：

  Agent接到task
      │
      ▼
  检索skill bank                          ← ❌ 不调API（embedding相似度计算，本地做）
      │
      ▼
  把skill拼入prompt + task描述
      │
      ▼
  Agent执行任务                            ← ✅ 调API（模型推理，生成action/CoT）
      │                                        这是主要开销
      │                                        ALFWorld: 每步调1次，平均~15步
      │                                        WebShop: 每步调1次，平均~10步
      │                                        MATH/BBH: 调1次生成完整CoT
      ▼
  任务结果
      │
      ├─ 成功 →
      │     判断覆盖率                     ← ✅ 调API（成功且用了skill时）
      │     Segmentation                   ← ✅ 调API（成功且需要收集时）
      │     PatternBuffer匹配              ← ❌ 不调API（embedding匹配）
      │     Formalization                  ← ✅ 调API（极少，满足提取条件时）
      │     Forgotten去重                  ← ❌ 不调API（embedding匹配）
      │
      └─ 失败（执行步数≥2步）→
            失败原因分析                   ← ✅ 调API（LLM分析失败原因）
            Warning匹配/转化               ← ❌ 不调API（embedding匹配已有skill）

  汇总

  ┌──────────────────────────┬──────────┬───────────────────────────┐
  │ 步骤                      │ 调API？  │ 频率                       │
  ├──────────────────────────┼──────────┼───────────────────────────┤
  │ Skill检索                 │ ❌       │ -                          │
  │ Agent执行任务              │ ✅       │ 每个task都调，主要开销      │
  │ 判断覆盖率                 │ ✅       │ 成功且用了skill时           │
  │ Segmentation              │ ✅       │ 成功且需要收集时            │
  │ PatternBuffer匹配         │ ❌       │ -                          │
  │ Formalization             │ ✅       │ 极少，满足提取条件时        │
  │ Forgotten去重             │ ❌       │ -                          │
  │ 失败原因分析               │ ✅       │ 失败且执行步数≥2步时           │
  │ Warning匹配               │ ❌       │ -（embedding匹配）         │
  ├──────────────────────────┼──────────┼───────────────────────────┤
  │ Meta更新                  │ ❌       │ -（更新计数器）              │
  │ Importance Score          │ ❌       │ -（公式计算）               │
  │ Merge相似度扫描            │ ❌       │ -（embedding计算）          │
  │ Distill（压缩skill描述）   │ ✅       │ 极少，超预算时才触发        │
  │ Archive压缩               │ ✅       │ 极少，降级时才触发          │
  └──────────────────────────┴──────────┴───────────────────────────┘
