================================================================================
                        Active Area (Skill Bank) 详解
================================================================================

定位

  Active = Skill Bank = agent检索和使用skill的唯一来源

  只有Active中的skill参与检索，Archive和Forgotten不参与。
  Active有token预算控制，通过Compression和Forgetting维持规模。


================================================================================
一、数据结构
================================================================================

  @dataclass
  class Skill:
      skill_id: str
      name: str
      description: str
      precondition: str
      parameters: List[str]
      steps: List[str]
      warnings: List[str]                   # 从失败轨迹中学到的
      source: str                           # "experience_distillation" 或 "failure_analysis"
      task_type: str                        # 来源任务类别
      confidence: float                     # 来自Alignment，持续更新
      token_cost: int                       # 这个skill占用的token数
      embedding: List[float]                # description的向量，用于检索

  @dataclass
  class SkillMeta:
      call_count: int                       # 总调用次数
      success_count: int                    # 使用该skill后任务成功的次数
      success_rate: float                   # success_count / call_count
      last_used_at: int                     # 最近一次使用的round
      avg_reward: float                     # 使用该skill时的平均task reward
      similarity_neighbors: List[str]       # 语义相似的skill ids

  @dataclass
  class ActiveSkill:
      skill: Skill
      meta: SkillMeta
      compressed: bool                      # 是否已被Distill压缩过
      version: int                          # 版本号（每次Merge/Distill +1）


================================================================================
二、Active里发生三件事
================================================================================

  ┌────────────────────────────────────────────────────────────┐
  │                     ACTIVE (Skill Bank)                      │
  │                                                              │
  │   ① 检索与使用 — agent每轮从这里取skill                      │
  │   ② Compression — 控制规模（Merge + Distill）                │
  │   ③ 降级判定   — importance低的skill移到Archive              │
  │                                                              │
  └────────────────────────────────────────────────────────────┘


================================================================================
三、① 检索与使用
================================================================================

--------------------------------------------------------------------------------
检索流程
--------------------------------------------------------------------------------

  Agent接到task
      │
      ▼
  task指令 → embedding
      │
      ▼
  和Active中每个skill的embedding算cosine similarity
      │
      ▼
  取top-K个（如K=3）且similarity > threshold的skill
      │
      ▼
  拼入agent的prompt

--------------------------------------------------------------------------------
拼入prompt的格式
--------------------------------------------------------------------------------

  你是一个完成{任务类型}的agent。

  以下是可能有用的技能：

  [Skill 1: 解二次方程]
  步骤：
    1. 确认方程为ax²+bx+c=0的形式
    2. 优先尝试因式分解
    3. 因式分解不行则用配方法或求根公式
    4. 代回原方程验证
  注意：
    - 系数简单时优先因式分解，不要直接用求根公式

  [Skill 2: 识别问题类型]
  步骤：
    1. 读题，识别关键数学对象
    2. 判断属于哪类问题

  当前任务：
  {task_instruction}

  请完成任务。

--------------------------------------------------------------------------------
使用后更新meta
--------------------------------------------------------------------------------

  Agent用了Skill S完成任务
      │
      ├─ 成功 →
      │     S.meta.call_count += 1
      │     S.meta.success_count += 1
      │     S.meta.success_rate = success_count / call_count
      │     S.meta.last_used_at = current_round
      │     S.meta.avg_reward = 更新running average
      │
      └─ 失败 →
            S.meta.call_count += 1
            S.meta.success_rate = success_count / call_count
            S.meta.last_used_at = current_round
            S.meta.avg_reward = 更新running average

  如果agent检索到了skill但没选用（在prompt里但agent忽略了）：
    → 不更新该skill的meta（不算一次调用）

--------------------------------------------------------------------------------
怎么知道agent有没有用检索到的skill
--------------------------------------------------------------------------------

  交互式任务（ALFWorld/WebShop）：
    检查trajectory中的action是否和skill的steps对应
    用覆盖率判断方法（LLM或embedding匹配）

  推理任务（MATH/BBH）：
    检查CoT中是否出现了skill描述的方法/步骤
    例：skill是"因式分解"，CoT中出现了因式分解 → 用了


================================================================================
四、② Compression — 控制规模
================================================================================

  目的：控制Active的总token在预算内。

--------------------------------------------------------------------------------
触发条件（任一满足）
--------------------------------------------------------------------------------

  1. 新skill从Acquisition进入Active → 检查是否超预算
  2. 定期维护（每N_maintain轮，如每100轮）→ 检查是否超预算

--------------------------------------------------------------------------------
三个操作，按优先级执行
--------------------------------------------------------------------------------

  超预算了
      │
      ▼
  优先级1: Merge（减少skill数量，信息损失最小）
      │
      ▼
  还超？
      │
      ├─ No → 结束
      └─ Yes ↓

  优先级2: Distill（减少单个skill的token，有少量信息损失）
      │
      ▼
  还超？
      │
      ├─ No → 结束
      └─ Yes ↓

  优先级3: Forgetting（降级importance最低的skill到Archive，最后手段）


================================================================================
Merge（合并相似skill）
================================================================================

--------------------------------------------------------------------------------
合并条件
--------------------------------------------------------------------------------

  条件1: 描述相似度 > τ_merge (如0.8)
    sim(S_i.embedding, S_j.embedding) > 0.8

  条件2: 来自同一类task_type 或 steps结构相似
    防止描述碰巧相似但实际用途不同的skill被错误合并

  两个条件都满足才Merge。

--------------------------------------------------------------------------------
Merge怎么做
--------------------------------------------------------------------------------

  Skill A: "在Amazon搜索商品并排序"  token_cost=80
  Skill B: "在eBay搜索商品并排序"    token_cost=85

  合并为：
  Skill C: "在{platform}搜索商品并排序"  token_cost=90
    - A标记为composed, 从Active移除
    - B标记为composed, 从Active移除
    - C进入Active
    - C.meta = 合并A和B的meta
        call_count = A.call_count + B.call_count
        success_rate = 加权平均
    - C.version = 1
    - C.warnings = A.warnings + B.warnings（去重）

  效果：2个skill(165 tokens) → 1个skill(90 tokens)，省了75 tokens

--------------------------------------------------------------------------------
用LLM做Merge
--------------------------------------------------------------------------------

  Prompt:
    以下两个skill功能相似，请合并为一个更通用的skill：

    Skill A: {skill_a}
    Skill B: {skill_b}

    要求：
    1. 合并后的skill要覆盖两者的功能
    2. 用参数化表示差异部分
    3. 合并warnings（去重）
    4. 描述精炼，控制token

--------------------------------------------------------------------------------
扫描策略
--------------------------------------------------------------------------------

  Active中skill数量通常不超过100个。
  直接扫描所有pair：100×99/2 = 4950次cosine similarity计算。
  embedding的cosine similarity计算很轻量，不调API，可以接受。

  如果后续skill数量增长到很大，可以优化为：
    每个skill维护similarity_neighbors（最相似的K个skill）
    只检查neighbor pair是否超过τ_merge
    新skill进入时更新neighbors
  但当前规模下不需要。


================================================================================
Distill（精炼单个skill）
================================================================================

--------------------------------------------------------------------------------
选谁来Distill
--------------------------------------------------------------------------------

  候选 = Active中所有 compressed == False 的skill（还没压缩过的）
  排序 = 按token_cost降序（先压最大的，效果最明显）
  逐个Distill，直到总token ≤ budget

  已经compressed == True的skill不再Distill（二次压缩信息损失大）。

--------------------------------------------------------------------------------
Distill怎么做
--------------------------------------------------------------------------------

  原始skill (200 tokens):
    steps: [
      "1. 首先，仔细阅读题目，确认方程是标准的二次方程形式ax²+bx+c=0",
      "2. 观察系数a、b、c的特点，如果系数是简单的小整数...",
      "3. 尝试因式分解：找两个数，它们的乘积等于ac，和等于b...",
      "4. 如果因式分解困难，考虑使用配方法...",
      "5. 最后的选择是直接套用求根公式...",
      "6. 求解完毕后，一定要将答案代回原方程进行验证..."
    ]

  Distill后 (80 tokens):
    steps: [
      "1. 确认ax²+bx+c=0形式",
      "2. 优先因式分解（找乘积=ac，和=b的两数）",
      "3. 不行则配方法或求根公式",
      "4. 代回验证"
    ]

  更新：
    skill.token_cost = 80
    skill.compressed = True
    skill.version += 1

--------------------------------------------------------------------------------
用LLM做Distill
--------------------------------------------------------------------------------

  Prompt:
    以下skill描述过长，请精炼：

    {skill}

    要求：
    1. 保留核心步骤和关键信息
    2. 去除冗余描述和解释性文字
    3. 目标token数: {target_tokens}
    4. 不要丢失warnings

--------------------------------------------------------------------------------
如果Merge + Distill都不够
--------------------------------------------------------------------------------

  触发Forgetting：降级importance最低的skill到Archive。
  见下面"③ 降级判定"。


================================================================================
五、③ 降级判定（Forgetting）
================================================================================

--------------------------------------------------------------------------------
Importance Score
--------------------------------------------------------------------------------

  每轮结束后，计算每个Active skill的importance score：

  I(s, t) = w1·Recency + w2·Frequency + w3·Quality + w4·Irreplaceability

  Recency:
    exp(-λ × (current_round - s.meta.last_used_at))
    越久没用 → 越低

    例：λ=0.1
      last_used = 5轮前  → exp(-0.5)  = 0.61
      last_used = 20轮前 → exp(-2.0)  = 0.14
      last_used = 50轮前 → exp(-5.0)  = 0.007

  Frequency:
    log(1 + s.meta.call_count)
    用得越多 → 越高（对数缩放，避免高频skill过度主导）

    例：
      call_count=0   → 0
      call_count=10  → 2.4
      call_count=100 → 4.6

  Quality:
    s.meta.success_rate × s.meta.avg_reward
    成功率和收益越高 → 越高

  Irreplaceability:
    1 - max(sim(s, s') for s' in Active, s'≠s)
    替代品越少 → 越高

    例：
      Skill A和B相似度0.9 → A的Irreplaceability = 0.1（很容易被B替代）
      Skill C和所有skill最高相似度0.3 → C的Irreplaceability = 0.7（独特，不可替代）

  Irreplaceability的意义：
    Skill A: 其他指标低(0.3)，但它是唯一能处理PDF的skill → 不应淘汰
    Skill B: 其他指标高(0.5)，但有3个功能相同的skill → 可以淘汰

--------------------------------------------------------------------------------
降级决策
--------------------------------------------------------------------------------

  每轮检查所有Active skill：

  if I(s, t) < θ_archive 连续T1轮:
      → 降级到Archive
      → 压缩skill描述后存入Archive
      → 从Active移除
      → 保留原始完整版本引用（Archive召回时恢复）

  为什么要"连续T1轮"：
    防止短暂波动导致误杀。
    某skill可能因任务类型暂时变化而importance下降，
    但很快又被需要。等T1轮再决定更稳妥。

--------------------------------------------------------------------------------
两种触发Forgetting的场景
--------------------------------------------------------------------------------

  场景1: 预算驱动（Compression的最后手段）
    Merge + Distill都做了还超预算
    → 强制降级importance最低的skill（不等T1轮）

  场景2: 自然衰退（定期检查）
    某skill长期不被使用，importance持续低于θ_archive
    → 连续T1轮后自然降级


================================================================================
六、一个完整的Active运作例子
================================================================================

  Round 1-50: 积累阶段
    Active逐渐积累skill
    没有Compression（还没超预算）
    没有Forgetting（都是新skill，importance高）
    Active: 15个skill, 总token=1200, 预算=2000

  Round 100: 持续积累
    Active: 28个skill, 总token=1950
    接近预算但没超

  Round 105: 2个新skill从Acquisition进入
    Active: 30个skill, 总token=2150 > 预算2000
    → 触发Compression

    Step 1 Merge扫描：发现S12和S18相似度0.87 > τ_merge
    → Merge → 29个skill, 总token=2050
    → 还超

    Step 2 Distill：S7的token_cost最高(180)，未压缩
    → Distill: 180→90 → 29个skill, 总token=1960
    → 在预算内了 ✓

  Round 150: 任务分布变化（Algebra → Geometry）
    Algebra相关skill不再被使用
    S1("因式分解") last_used=105

    Round 150: I(S1) = 0.28 < θ_archive(0.3) → 第1轮低于阈值
    Round 151: I(S1) = 0.25 → 第2轮
    Round 152: I(S1) = 0.22 → 第3轮 = T1
    → S1降级到Archive → 压缩存储 → 从Active移除
    → 28个skill

  Round 300: 任务分布回到Algebra
    Archive中S1被检索命中 → 召回 → 成功R次 → 回到Active
    重置meta，重新积累

================================================================================
八、超参数汇总
================================================================================

  参数                含义                              建议范围
  ─────────────────────────────────────────────────────────────────
  K                   检索返回的skill数                  2~5
  threshold           检索最低相似度                     0.3~0.5
  budget              Active总token预算                  实验确定
  τ_merge             Merge触发的相似度阈值              0.75~0.85
  θ_archive           降级到Archive的importance阈值      0.2~0.4
  T1                  连续低于阈值的轮数要求              3~5
  w1, w2, w3, w4      Importance score四项权重           实验确定
  λ                   Recency指数衰减系数                0.05~0.15
  N_maintain          定期维护间隔轮数                    50~100


================================================================================
九、调API的地方
================================================================================

  Active内部操作中需要调API的：

  ┌──────────────────────────┬──────────┬───────────────────────────┐
  │ 操作                      │ 调API？  │ 频率                       │
  ├──────────────────────────┼──────────┼───────────────────────────┤
  │ Skill检索(embedding匹配) │ ❌       │ -（本地cosine similarity） │
  │ Agent执行任务              │ ✅       │ 每个task都调，主要开销      │
  │ 判断agent是否用了skill    │ ✅       │ 每个成功task（可用规则替代）│
  │ Meta更新                  │ ❌       │ -（更新计数器）              │
  │ Importance Score计算      │ ❌       │ -（公式计算）               │
  │ Merge相似度扫描            │ ❌       │ -（embedding cosine）       │
  │ Merge执行（LLM合并描述）   │ ✅       │ 极少，超预算且有相似pair时  │
  │ Distill执行（LLM精炼描述） │ ✅       │ 极少，超预算时              │
  └──────────────────────────┴──────────┴───────────────────────────┘

  90%以上的API开销在"Agent执行任务"。
  Active内部管理操作的API调用极少。
