================================================================================
        Skill Lifecycle Management Framework: Skill Bank as a Living Ecosystem
================================================================================

核心理念：Skill bank不是仓库（warehouse），而是生态系统（ecosystem）。
借鉴人类认知科学的记忆理论，模拟技能的完整生命周期。

  Skill Bank = Active Area（唯一的检索来源）
  Archive = 曾经有用但当前不活跃的skill的缓冲区
  Forgotten = 去重黑名单，防止重复学习

================================================================================
一、Acquisition — 增量式Experience Distillation
================================================================================

  选择Experience Distillation作为获取方式。
  理由：自包含、有清晰baseline（Voyager/SAGE）、novelty在lifecycle不在acquisition。

  两类数据集的原材料不同，但统一为step序列：
    ALFWorld/WebShop: action trajectory（操作序列）
    MATH/BBH:         reasoning chain（推理链CoT）

--------------------------------------------------------------------------------
收集判断
--------------------------------------------------------------------------------

  Agent完成task
      │
      ├─ 成功 → 成功路径（提取skill的steps）
      │    │
      │    ├─ 没用任何skill（裸跑成功）→ 收集整条trajectory
      │    │
      │    └─ 用了skill成功 → LLM判断覆盖率
      │         │
      │         ├─ 覆盖率100% → 不收集，只更新meta
      │         ├─ 覆盖率>50%  → 只收集未覆盖的片段
      │         └─ 覆盖率≤50%  → 收集整条trajectory
      │
      └─ 失败 → 失败路径（提取warnings）
           │
           ├─ 执行步数 < 2步 → 跳过
           │
           └─ 执行步数 ≥ 2步 → LLM分析失败原因 → 提取warning
                │
                ├─ 匹配到已有Active skill → 附加到该skill的warnings
                │
                └─ 没匹配到 → 有行动指导？
                     │
                     ├─ 有 → 转化为独立skill（confidence低）进Active
                     └─ 没有 → 丢弃

--------------------------------------------------------------------------------
Segmentation（切分）
--------------------------------------------------------------------------------

  交互式任务（ALFWorld/WebShop）：按环境状态变化切分
  推理任务（MATH/BBH）：按推理阶段切分
  统一用LLM做切分

--------------------------------------------------------------------------------
Incremental Alignment（增量式跨记录对齐）
--------------------------------------------------------------------------------

  前提：知道哪些task属于同一类
    ALFWorld: 自带标签（6种任务类型）
    MATH:     自带标签（7个类别×5个难度）
    BBH:      自带标签（23种任务类型）
    WebShop:  无标签，需要LLM打标签或embedding聚类

  核心数据结构：PatternBuffer（每类任务维护一个）

  每来一条新的成功记录：
    1. Segmentation切分为片段
    2. 放入对应task_type的PatternBuffer
    3. 和已有patterns语义匹配，更新count和confidence
    4. 检查提取条件：
       total_records ≥ M 且 confidence ≥ r 且 未被提取过 且 非跨类别通用
    5. 满足 → Formalization → 查Forgotten去重 → 进Active

  Confidence持续更新：
    已提取的skill的confidence随新记录动态变化
    更新传递到Active中对应skill的meta

--------------------------------------------------------------------------------
Skill格式
--------------------------------------------------------------------------------

  {
    "name": "quadratic_equation_solver",
    "description": "识别二次方程并选择合适的解法",
    "precondition": "表达式包含x²项",
    "parameters": ["method: factoring | completing_square | quadratic_formula"],
    "steps": [
      "1. 确认方程为ax²+bx+c=0的形式",
      "2. 优先尝试因式分解（简单情况）",
      "3. 因式分解不行则用配方法或求根公式",
      "4. 代回原方程验证"
    ],
    "warnings": [
      "系数简单时优先因式分解，不要直接用求根公式（容易算错）"
    ],
    "source": "experience_distillation",
    "task_type": "algebra_level3",
    "confidence": 0.8,
    "token_cost": 95
  }


================================================================================
二、Active Area（Skill Bank）
================================================================================

  Active = Skill Bank = agent检索和使用skill的唯一来源

--------------------------------------------------------------------------------
检索与使用
--------------------------------------------------------------------------------

  每个task开始时检索一次（方案A，和SkillRL一致）：
    task指令 → embedding → 和Active中每个skill算cosine similarity
    → 取top-K（如K=3）且sim > threshold → 拼入prompt
    → agent带着这些skill执行整个任务

  任务结束后更新已用skill的meta：
    call_count, success_count, success_rate, last_used_at, avg_reward

  如果Active中最高sim < threshold → 搜索Archive

--------------------------------------------------------------------------------
Compression（超预算时触发）
--------------------------------------------------------------------------------

  触发条件：新skill进Active后超预算 或 定期维护（每N_maintain轮）

  优先级1: Merge（合并相似skill）
    条件：sim > τ_merge 且 同一task_type或steps结构相似
    用LLM合并为参数化skill，合并meta和warnings

  优先级2: Distill（精炼长skill）
    候选：compressed == False的skill，按token_cost降序
    用LLM精炼，压缩后标记compressed = True
    已压缩的不再二次Distill

  优先级3: Forgetting（降级importance最低的skill到Archive）

--------------------------------------------------------------------------------
Importance Score
--------------------------------------------------------------------------------

  I(s,t) = w1·Recency + w2·Frequency + w3·Quality + w4·Irreplaceability

  Recency:          exp(-λ × (current_round - last_used_at))
  Frequency:        log(1 + call_count) / log(1 + max_count)  （归一化到[0,1]）
  Quality:          success_rate × avg_reward
  Irreplaceability: 1 - max(sim(s, s') for s' in Active, s'≠s)

  连续T1轮低于θ_archive → 降级到Archive


================================================================================
三、Archive Area
================================================================================

  唯一入口: Active降级
  存储: 压缩版summary + 完整版备份（original_skill_full）
  不参与正常检索

--------------------------------------------------------------------------------
触发检索
--------------------------------------------------------------------------------

  Active中检索不到合适skill（最高sim < threshold）
  → 搜索Archive
  → 用task embedding和Archive中skill_summary的embedding匹配

--------------------------------------------------------------------------------
召回流程
--------------------------------------------------------------------------------

  Archive skill被命中 → 恢复完整版 → 拼入prompt → 执行任务
    成功 → recall_count += 1, inactive_rounds = 0
    失败 → 留在Archive, inactive_rounds = 0（有人需要，重置计数）

  recall_count ≥ R → Promote回Active（重置meta）

--------------------------------------------------------------------------------
通向Forgotten
--------------------------------------------------------------------------------

  连续max_archive_inactive轮无人召回 → 降级到Forgotten
  丢弃完整版本，只保留{skill_id, name, summary, forgotten_at}


================================================================================
四、Forgotten Area
================================================================================

  唯一入口: Archive降级
  存储: 只有 {skill_id, name, summary, forgotten_at}
  不参与检索

  唯一用途: 新skill进Active前做去重检查

  去重逻辑：
    新skill和Forgotten中某skill相似（sim ≥ τ）
    且淘汰时间 < D轮 → 跳过（最近才淘汰，大概率还是没用）
    且淘汰时间 ≥ D轮 → 允许进Active（任务分布可能变了）

  Trajectory/CoT
      │

七、实验设计
================================================================================

  核心story：同样的acquisition输入，不同的lifecycle策略，performance差距显著。

  ┌──────────────────────────┬───────────┬───────────┬──────────┬─────────┐
  │ 实验                      │ ALFWorld  │ WebShop   │ MATH     │ BBH     │
  ├──────────────────────────┼───────────┼───────────┼──────────┼─────────┤
  │ 1. Scaling Analysis      │           │     ✅     │    ✅     │         │
  │ 2. Consolidation Ablation│           │     ✅     │          │         │
  │ 3. Compression Ablation  │           │     ✅     │          │         │
  │ 4. Forgetting策略对比     │     ✅     │           │          │         │
  │ 5. 闭环 vs. 单机制        │     ✅     │           │    ✅     │         │
  │ 6. Non-Stationary        │     ✅     │           │    ✅     │   ✅    │
  └──────────────────────────┴───────────┴───────────┴──────────┴─────────┘

  实验1: Skill Bank Scaling Analysis
    固定其他变量，只变budget，画performance-size曲线，找sweet spot

  实验2: Consolidation Ablation
    有Acquisition过滤 vs 无过滤（Voyager式直接存储）

  实验3: Compression Ablation
    Full(Merge+Distill) vs Merge-Only vs Distill-Only vs No-Compress

  实验4: Forgetting策略对比
    Importance-based vs LRU vs LFU vs Random vs No-Forget

  实验5: 闭环 vs. 单机制
    完整lifecycle vs 只有其中一个机制，证明协同效应

  实验6: Non-Stationary Task Stream
    任务分布按phase轮换，测试Archive召回能力
    Phase 4回归是最强selling point

  Baseline:
    No Skill | Voyager（只增不减）| FIFO | LRU | Ours
